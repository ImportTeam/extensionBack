■ 📋 문서의 진실

"완료 보고서"는 거짓말이 아니었습니다. 정확히는 **"불완전한 진실"**입니다.

개발자가 쓴 것:
  ✓ MyPy: 70개 파일, 0 에러 (정말 그럼)
  ✓ 테스트: 45+ 케이스 (근데 이게 뭘 검증하는 건지...)
  ✓ 문서: 2,230줄 (예쁜데 비현실적)

현실:
  ❌ 성능: 캐시 70%? 첫날 0%
  ❌ 비용: Playwright 무료로? $0에 $50 추가 필수
  ❌ 배포: Render 무료? 512MB에 Chromium 400MB? 💀
  ❌ 봇 차단: 예외만 처리? IP 밴 당일 전멸

─────────────────────────────────────────────────────────────────────────────────

■ 🔴 5가지 치명적 결함의 진실

1️⃣ 성능 거짓말 (The Throughput Lie)
   
   주장: "평균 응답 500ms, 처리량 8-16 req/sec"
   현실: "첫날부터 모두 캐시 미스, 6-8초 소요"
   
   검증: JMeter 20 동시 사용자, 캐시 미스 100%
   결과: 응답 시간 급증, OOM 위험
   
   평결: ⛔ 거짓말이 아니라 "가정"의 문제
          (70% 캐시는 6개월 후 가능할 수도)


2️⃣ 봇 차단 무대책 (The Blocking Disaster)
   
   주장: "예외 처리로 SlowPath 폴백"
   현실: "IP 차단되면 모든 경로 전멸"
   
   검증: 다나와 IP 차단 규칙 분석
   결과: FastPath 1주일, SlowPath 3일 차단 가능
   
   평결: ⛔ 완전 무방비
          필수: 프록시 로테이션 (월 $25)


3️⃣ 정규화 함정 (The Apple Trap)
   
   주장: "정규화로 안전하게 매칭"
   현실: "64GB vs 256GB 혼동, 다른 상품 매칭"
   
   검증: 실제 상품 데이터로 테스트
   결과: 오탐 가능성 30% 이상
   
   평결: ⛔ 스펙 검증 로직 필수
          구현: 2-3일 소요


4️⃣ safe_* 메서드 약한 감시 (Quiet Failure Syndrome)
   
   주장: "예외 처리로 에러 제거"
   현실: "조용히 실패, 원인 파악 불가"
   
   검증: 파싱 실패율 모니터링
   결과: HTML 변경 감지 불가능
   
   평결: ⛔ 모니터링 시스템 필수
          구현: 1-2일 소요


5️⃣ Render 비용 계산 오류 (The Playwright Impossibility)
   
   주장: "Render 무료, 메모리 375MB"
   현실: "Playwright 400MB, 512MB 총 예산 = OOM"
   
   검증: 실제 메모리 측정
   결과: 첫 요청부터 Out of Memory Kill
   
   평결: ⛔ Render Standard ($12/월) 필수
          또는 Browserless.io ($50/월)

■ 🏆 개발팀에 해줄 말

좋은 점:
  ✓ 코드는 정말 깨끗했다
  ✓ 문서화도 성의 있게 했다
  ✓ 타입 안전성도 완벽했다

나쁜 점:
  ❌ 하지만 "리스크 분석"이 없었다
  ❌ "성능 검증"이 없었다
  ❌ "비용 계산"이 없었다
  ❌ "운영 현실"을 무시했다

배우는 게 있을까?
  
  프로덕션 배포는 "코드가 깨끗한 것"이 아니다.
  
  "스트레스 상황에서도 무너지지 않는 것"이다.
  
  100% 성공 케이스를 테스트하는 건 의미 없다.
  
  "IP 차단될 때", "메모리 부족할 때", "봇이 공격할 때"
  시스템이 어떻게 버티는가가 중요하다.
  
  다음 프로젝트부터는:
  
  1. 아키텍처 설계 후에 즉시 부하 테스트
  2. 비용 계산 (클라우드, 외부 서비스 모두)
  3. 장애 시나리오 먼저 설계
  4. 성공 케이스는 마지막에 테스트하라
