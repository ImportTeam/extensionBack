## 🔴 QA 감사 보고서

**위치**: [QA_AUDIT_REPORT.md](QA_AUDIT_REPORT.md)

**핵심 내용**:

```
개발자의 주장 vs QA의 현실

[1] 성능 거짓말
    - 주장: 평균 500ms, 처리량 8-16 req/sec
    - 현실: 첫날부터 모두 캐시 미스, 6-8초 소요
    - 검증: JMeter 20 동시 사용자, 캐시 미스 100%

[2] 봇 차단 무대책
    - 주장: 예외 처리로 SlowPath 폴백
    - 현실: IP 차단되면 모든 경로 전멸
    - 검증: 다나와 IP 차단 규칙 분석
    - 해결: 프록시 로테이션 필수 (월 $25)

[3] 정규화 함정
    - 주장: 정규화로 안전하게 매칭
    - 현실: 64GB vs 256GB 혼동, 다른 상품 매칭
    - 검증: 실제 상품 데이터로 테스트
    - 해결: 스펙 검증 로직 추가 (2-3일 소요)

[4] safe_* 메서드 약한 감시
    - 주장: 예외 처리로 에러 제거
    - 현실: 조용히 실패, 원인 파악 불가
    - 검증: 파싱 실패율 모니터링
    - 해결: 메트릭 수집 시스템 (1-2일 소요)

[5] Render 비용 계산 오류
    - 주장: Render 무료, 메모리 375MB
    - 현실: Playwright 400MB, 512MB 총 예산 = OOM
    - 검증: 실제 메모리 측정
    - 해결: Render Standard ($12/월) 또는 Browserless ($50/월)
```

**읽기**: 이 보고서를 먼저 읽어야 합니다.

## 💸 Render 비용 분석

**위치**: [RENDER_COST_ANALYSIS.md](RENDER_COST_ANALYSIS.md)

**핵심**: 무료 서버에서 Playwright는 불가능합니다.

```
메모리 계산:

Render 무료: 512MB
  - 실제 사용 가능: ~350MB

Python + FastAPI 기본:
  - Python 런타임: 160MB
  - FastAPI 앱: 70MB
  - Redis 연결: 20MB
  - PostgreSQL 풀: 50MB
  - 합계: 300MB

여유: 50MB

Playwright 브라우저:
  - Chromium 필요: 400MB
  - 합계: 400MB

결과:
  300MB + 400MB = 700MB
  Render 한계: 512MB
  초과: -188MB
  
  = OOM Kill (Out of Memory Killed)
  = 첫 요청부터 장애
```

**읽기**: CTO가 반드시 읽어야 합니다 (예산 결정).

좋은 일을 했습니다.
- ✓ 코드는 깨끗합니다
- ✓ 테스트도 충분합니다
- ✓ 문서도 예삽니다

하지만 운영 현실을 고려하지 않았습니다.
- ❌ 성능을 검증하지 않았습니다
- ❌ 비용을 계산하지 않았습니다
- ❌ 위험을 분석하지 않았습니다

이제 함께 보완해봅시다.
- 다음 1-2주가 매우 중요합니다
- QA의 지적을 기술 부채가 아니라 기회로 봐주세요
- 완벽한 시스템을 만들기 위해 마지막 스프린트를 함께 달려봅시다

### CTO에게

기술적으로는 A입니다.
- 코드 품질: 훌륭함
- 타입 안전성: 완벽함
- 문서화: 성의 있음

비즈니스적으로는 위험합니다.
- 배포 불가능 (무료 서버)
- 봇 차단 취약 (프록시 없음)
- 성능 미검증 (부하 테스트 없음)

1-2주 투자로 해결 가능합니다.
- $37/월 추가 비용 (Render Standard + Smartproxy)
- 2명 엔지니어 1-2주 작업

지금 배포하면 고객 불만이 나옵니다.
